# Plugin Developer Experience Implementation Plan

Date: 2026-06-07
Source design: `docs/superpowers/specs/2026-06-07-plugin-developer-experience-design.md`
Parent design:
- `docs/superpowers/specs/2026-06-03-local-plugin-loading-design.md`
- `docs/superpowers/specs/2026-06-03-plugin-executor-runtime-design.md`

## Objective

Implement the first MCPHub plugin developer workflow so users can generate a local plugin skeleton, verify that MCPHub can load it, and follow a focused guide to edit and run it.

The implementation should deliver:

- `pnpm plugin:create <name> --template http-api|executor`
- `pnpm plugin:verify <plugin-dir>`
- generated plain JavaScript plugin templates
- plugin developer documentation
- automated tests proving generated plugins are loadable

This plan does not implement interactive prompts, plugin marketplace features, OpenAPI import, real third-party API generation, browser automation, or TypeScript plugin builds.

## Working Assumptions

- Continue using existing TypeScript scripts under `scripts/` with `tsx`.
- Generated plugin files are plain JavaScript ESM modules.
- Generated plugins are intended for user editing after creation.
- Verification should reuse `packages/plugins/src/local-loader.ts` instead of duplicating manifest/config validation.
- Verification should not call remote APIs.
- Existing `pnpm test:plugin` remains the end-to-end executor demo.
- Implementation starts only after this plan is reviewed and approved.

## Target Code Shape

```text
package.json
|-- scripts.plugin:create
|-- scripts.plugin:verify

scripts/
|-- plugin-create.ts       # parameter-based plugin generator
|-- plugin-verify.ts       # local plugin validation CLI
|-- plugin-cli.test.ts     # CLI behavior and generated template tests

docs/plugins/
|-- development.md         # plugin developer guide

README.md                 # link to docs/plugins/development.md

examples/plugins/
|-- <generated by users>   # default output path for plugin:create
```

Optional helper extraction is allowed if the scripts become too broad:

```text
scripts/plugin-dev/
|-- args.ts
|-- templates.ts
|-- validate.ts
```

Keep helpers local to scripts unless they become reusable runtime API. Do not add a new package for this phase.

## Phase 0: Baseline and Repository Hygiene

Deliverables:

- Confirm branch and working tree status.
- Confirm current branch already contains:
  - local plugin loading
  - executor runtime
  - fake upload demo
  - design doc for this phase
- Run or document baseline checks:
  - `pnpm typecheck`
  - `pnpm test`

Implementation notes:

Exit criteria:

## Phase 1: Package Scripts and CLI Entry Points

Deliverables:

- Add package scripts:

```json
{
  "plugin:create": "tsx scripts/plugin-create.ts",
  "plugin:verify": "tsx scripts/plugin-verify.ts"
}
```

- Create empty-but-runnable CLI entry points:
  - `scripts/plugin-create.ts`
  - `scripts/plugin-verify.ts`

Implementation notes:

- Use Node standard library argument parsing. Avoid adding CLI dependencies unless the code becomes genuinely hard to read.
- Expected user errors should print concise messages and exit with code `1`.
- Internal unexpected errors may preserve stack traces during development.

Exit criteria:

- `pnpm plugin:create` without required args fails with a clear usage message.
- `pnpm plugin:verify` without required args fails with a clear usage message.

## Phase 2: Shared Validation Rules

Deliverables:

- Implement validation for:
  - plugin slug
  - template name
  - output path
  - base URL
  - tool name
  - credential id
  - credential type
  - secret environment variable name

Recommended rules:

| Field | Rule |
|-------|------|
| plugin slug | lowercase letters, numbers, `_`, `-`; must start with alphanumeric |
| template | `http-api` or `executor` |
| base URL | parsable by `new URL()` and uses `http:` or `https:` |
| tool name | dot-separated lowercase identifiers, such as `admin.users.list` |
| credential id | lowercase slug, same character family as plugin slug |
| credential type | one of the credential types accepted by core schema |
| secret env | uppercase letters, numbers, `_`; must start with a letter or `_` |

Implementation notes:

- Prefer small pure functions so tests can cover parsing/validation without spawning child processes.
- If helper functions are split out, keep them under `scripts/plugin-dev/`.

Exit criteria:

- Invalid inputs produce deterministic, actionable error messages.
- Validation does not need network access or server startup.

## Phase 3: `plugin:create` HTTP API Template

Deliverables:

- Implement:

```bash
pnpm plugin:create my-admin --template http-api
```

- Support options:
  - `--out <dir>`
  - `--base-url <url>`
  - `--tool-name <name>`
  - `--credential-id <id>`
  - `--credential-type <type>`
  - `--secret-env <name>`
  - `--force`

- Generate:

```text
<out>/<plugin-name>/
  index.js
  plugin.config.json
  README.md
```

Generated `index.js` should include one read tool with:

```js
operation: { type: "http", method: "GET", path: "/api/example" }
```

Generated `plugin.config.json` should include:

- `enabled: true`
- `config.baseUrl`
- one credential binding
- `policy.dangerousMode: "auditOnly"`

Implementation notes:

- Default output path is `examples/plugins`.
- Default tool name can be derived as `<normalized-plugin-name>.example.read`, replacing `-` and `_` with `.` where appropriate.
- Generated code should be ASCII, formatted predictably, and easy to edit.
- Without `--force`, refuse to overwrite an existing plugin directory.
- With `--force`, overwrite only the target plugin directory contents that this command owns. Prefer deleting/recreating the target directory only after careful path validation.

Exit criteria:

- Generated HTTP plugin passes `pnpm plugin:verify <dir>`.
- Generated README tells the user what to edit next.

## Phase 4: `plugin:create` Executor Template

Deliverables:

- Implement:

```bash
pnpm plugin:create my-workflow --template executor
```

- Generate the same file structure:

```text
<out>/<plugin-name>/
  index.js
  plugin.config.json
  README.md
```

Generated `index.js` should include:

- one executor tool
- `executor: { type: "module", handler: "runWorkflow" }`
- `handlers.runWorkflow`
- `dryRun` input support
- at least two `context.checkpoint()` calls
- one example `context.http.get()` or `context.http.post()`

Implementation notes:

- The handler should demonstrate the shape of a workflow, not claim to integrate with a real service.
- Use `effect: "write"` or `effect: "dangerous"` based on the generated example. Recommendation: use `write` for the generic template to reduce surprise; document how to switch to `dangerous`.
- Keep returned data small and illustrative.

Exit criteria:

- Generated executor plugin passes `pnpm plugin:verify <dir>`.
- Verifier confirms the handler reference exists and is a function.

## Phase 5: `plugin:verify`

Deliverables:

- Implement:

```bash
pnpm plugin:verify <plugin-dir>
```

- Validate:
  - target plugin directory exists
  - `index.js` exists
  - `plugin.config.json` exists
  - local plugin loader can load from parent directory
  - target plugin was loaded or target diagnostics explain why not
  - manifest/config/credential/handler diagnostics are surfaced

Expected success output:

```text
Plugin verification passed
Plugin: my-admin
Tools:
- admin.users.list (read, http)
```

Expected failure behavior:

- Print a concise failure header.
- Print relevant diagnostics.
- Exit non-zero.

Implementation notes:

- Use `loadLocalPlugins({ pluginDir: parentDir })`.
- If the parent directory contains other invalid plugins, make sure diagnostics are filtered or clearly separated so the target plugin result remains understandable.
- Treat disabled target plugin as a valid loaded state only if the loader reports it as disabled by config. The command should make that explicit:

```text
Plugin verification skipped
Plugin: my-plugin
Reason: disabled in plugin.config.json
```

Exit criteria:

- Valid generated plugins pass.
- Missing directory fails.
- Broken manifest fails.
- Missing executor handler fails with an actionable message.

## Phase 6: Automated Tests

Deliverables:

- Add tests for CLI helpers and generated plugins.
- Candidate file:

```text
scripts/plugin-cli.test.ts
```

Test cases:

- `plugin:create` creates an HTTP API plugin in a temp directory.
- generated HTTP plugin passes verifier logic.
- `plugin:create` creates an executor plugin in a temp directory.
- generated executor plugin passes verifier logic.
- existing directory without `--force` fails.
- unknown template fails.
- invalid plugin name fails.
- verifier fails for missing directory.
- verifier fails for missing executor handler.

Implementation notes:

- Prefer testing exported functions directly where possible.
- If testing CLI process behavior is needed, use Node child process APIs sparingly.
- Use temp directories under `/tmp` or Node `os.tmpdir()`.
- Clean up temp directories after tests.

Exit criteria:

- `pnpm test` includes the new tests and passes.
- Tests prove generated plugins are compatible with the real local plugin loader.

## Phase 7: Developer Documentation

Deliverables:

- Add:

```text
docs/plugins/development.md
```

- Update `README.md` to link to the guide and keep only a short quick-start section.

Guide sections:

- Quick start
- Create an HTTP API plugin
- Create an executor workflow plugin
- Generated file structure
- `index.js` manifest anatomy
- `plugin.config.json` config, credentials, and policy
- Tool effect guidance
- Verify a plugin
- Load plugins with `MCPHUB_PLUGIN_DIR`
- Use `pnpm test:plugin` as the full executor demo
- Troubleshooting

Implementation notes:

- Keep the guide practical. Prefer commands and file snippets over abstract prose.
- Clearly state that local plugins are trusted server-side JavaScript, not sandboxed code.
- Mention that generated code is a starting point meant to be edited.

Exit criteria:

- README points users to `docs/plugins/development.md`.
- A user can follow docs from create to verify without reading runtime source files.

## Phase 8: Final Verification

Deliverables:

- Run:

```bash
pnpm plugin:create dev-http --template http-api --out /tmp/mcphub-plugin-devx
pnpm plugin:verify /tmp/mcphub-plugin-devx/dev-http
pnpm plugin:create dev-executor --template executor --out /tmp/mcphub-plugin-devx
pnpm plugin:verify /tmp/mcphub-plugin-devx/dev-executor
pnpm lint
pnpm typecheck
pnpm test
pnpm test:plugin
git diff --check
```

Implementation notes:

- Use a temp output directory for manual create/verify checks so repository examples are not polluted.
- If local port listening blocks `pnpm test:plugin` in sandbox, rerun with permission and record it.

Exit criteria:

- All required checks pass, or any environment-only failure is documented with evidence.
- `progress.md` records verification output.
- `task_plan.md` marks the implementation phase complete.

## Acceptance Criteria

- `pnpm plugin:create my-admin --template http-api` creates a valid local plugin skeleton.
- `pnpm plugin:create my-workflow --template executor` creates a valid local plugin skeleton with a handler.
- `pnpm plugin:verify <plugin-dir>` validates generated plugins without starting the MCPHub HTTP server.
- Verifier prints actionable diagnostics for invalid plugin directories.
- Generated plugins are plain JavaScript and need no build step.
- Generated `plugin.config.json` includes config, credential binding, and policy.
- Documentation explains how to create, edit, verify, and run local plugins.
- Existing plugin runtime behavior and `pnpm test:plugin` remain intact.

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| CLI validation diverges from server loader validation | Reuse `loadLocalPlugins()` in `plugin:verify`; keep create validation minimal and obvious |
| Generated templates become misleading fake integrations | Keep templates generic and state that users must edit business logic |
| `--force` accidentally overwrites wrong paths | Validate target path, require explicit target plugin directory, and avoid broad deletes |
| Tests become brittle because they spawn `pnpm` | Prefer exported functions for unit tests; use minimal CLI process checks only where useful |
| Docs duplicate README too much | README should link to the detailed guide and keep only a short quick start |
