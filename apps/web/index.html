Platform positioning

它不是单个插件，而是把外部能力转成 MCP 的中间平台。

MCPHub 的目标接近 RSSHub 的扩展模型：运行一个统一服务，让开发者用插件把不同网站、 API 和工作流接入到稳定的 Agent 接口。

01

不是单一业务插件

单个 B 站、后台或 SaaS 接入只是平台能力的样板，核心是可复制的插件标准。

02

不替代原后台

原系统继续使用自己的 API、权限和业务规则，MCPHub 只作为可信服务端适配层。

03

面向 Agent 的标准出口

插件能力最终暴露为 MCP resources 和 tools，让 Agent 能发现、理解和调用。

Current capabilities

当前 dev 版本已经具备完整的接入闭环。

从插件创建、加载、执行到 MCP client 验证，MCPHub 已经可以作为自托管开发版本使用。

本地插件加载

loader

通过 `MCPHUB_PLUGIN_DIR` 加载可信服务端 JavaScript 插件。

HTTP API tools

http

将一次 REST 请求映射成 Agent 可调用的 MCP tool。

Executor workflows

exec

用插件自定义 handler 执行多步骤上传、轮询、清洗和聚合。

凭据与策略

policy

从环境变量解析凭据，并区分 `read`、`write`、`dangerous` 效果。

Audit evidence

audit

记录工具调用、策略拦截和 executor checkpoint，便于审计和回滚。

Plugin standard

verify

通过插件标准、metadata 和 `plugin:verify` 提前发现接入问题。

MCP endpoint

/mcp

通过 Streamable HTTP 暴露 `resources/list`、`resources/read`、`tools/list`、`tools/call`。

Generic client CLI

client

用 `pnpm mcp:client` 直接检查 endpoint、资源、工具和调用结果。

Developer workflow

从一个 API 文档或后台能力开始，逐步变成 MCP tool。

首版路径保持朴素：生成插件骨架，人工补齐业务细节，验证插件，再启动 MCPHub 给 Agent 使用。

1

创建插件

选择 HTTP API 或 executor 模板，生成可编辑的本地插件目录。

2

验证插件

运行标准校验，提前发现 manifest、config、凭据和 tool 声明问题。

3

启动 MCPHub

加载插件目录，平台负责策略、审计、诊断和 MCP 暴露。

4

让 Agent 调用

通过标准 MCP endpoint 发现工具并执行调用。

terminal 
pnpm plugin:create my-admin --template http-api --tool-name admin.users.list
pnpm plugin:verify examples/plugins/my-admin
MCPHUB_PLUGIN_DIR=examples/plugins pnpm dev
pnpm mcp:client --url http://127.0.0.1:3000/mcp list-tools
Plugin model

插件和平台解耦，复杂业务放在插件执行器里。

简单 REST 操作用声明式 `operation`，复杂上传、校验、提交和轮询用 `executor`。 平台只提供受控上下文和统一的 MCP 出口。

examples/plugins/my-plugin/ |-- index.js |-- plugin.config.json |-- README.md
index.js 
export default {
  id: "admin-users",
  version: "0.1.0",
  type: "api",
  mcphub: {
    minVersion: "0.1.0",
    capabilities: ["http", "credentials", "policy"]
  },
  tools: [{
    name: "admin.users.list",
    effect: "read",
    operation: { type: "http", method: "GET", path: "/api/users" }
  }]
};
Status and roadmap

当前重点是把平台标准变得更可复制。

v0.1/dev 已经跑通基础平台链路，下一步会降低 API 文档转 MCPHub 插件的成本。

Available now

  • Dev server and Docker smoke checks
  • Local plugin loading and verification
  • HTTP tools and executor workflow tools
  • MCP status, plugin diagnostics, audit resources

Next

  • OpenAPI/API docs to plugin generator
  • Richer plugin lifecycle diagnostics
  • Deployment hardening for public endpoints
  • More repeatable developer contribution standards

Boundaries

  • No hosted cloud service is promised yet
  • No public plugin marketplace yet
  • No complete OpenAPI coverage yet
  • Local plugins are trusted server-side code
